# tiny-platform vs tiny-project 命名对比分析

## 一、项目内容分析

### 当前项目包含的模块

#### 业务应用模块
- `tiny-oauth-server` - OAuth2 授权服务器（应用）
- `tiny-oauth-client` - OAuth2 客户端（应用）
- `tiny-oauth-resource` - OAuth2 资源服务器（应用）
- `tiny_web` - Web 应用（应用）

#### 基础设施/平台模块
- `tiny-idempotent-platform` - 幂等平台（基础设施）
- `tiny-common-exception` - 通用异常处理（基础设施）
- `tiny-idempotent-starter` - 幂等 Starter（基础设施）

### 项目特征

1. **既有应用，也有基础设施**
   - 包含了多个业务应用（OAuth 相关）
   - 包含了基础设施组件（幂等平台、异常处理）

2. **多模块结构**
   - 是一个多模块 Maven 项目
   - 包含业务应用和基础设施的统一管理

## 二、命名对比分析

### 选项 A：`tiny-platform`

#### ✅ 优点

1. **语义准确** ⭐⭐⭐⭐⭐
   - "platform" 表示平台，包含多个应用和基础设施
   - 准确反映了项目的结构和定位
   - 表明这是一个提供多种服务的平台

2. **符合行业惯例** ⭐⭐⭐⭐
   - 类似项目：`spring-platform`、`netflix-platform`
   - 平台类项目通常使用 `-platform` 后缀

3. **未来扩展性好** ⭐⭐⭐⭐⭐
   - 如果未来增加更多基础设施组件（如限流平台、监控平台），名称仍然适用
   - 如果未来增加更多业务应用，名称仍然适用

4. **专业感** ⭐⭐⭐⭐
   - "platform" 听起来更专业，表明是一个企业级平台
   - 适合对外展示和推广

#### ⚠️ 缺点

1. **可能过于"宏大"** ⭐⭐
   - 如果项目规模较小，用 "platform" 可能显得过度设计

### 选项 B：`tiny-project`

#### ✅ 优点

1. **通用性强** ⭐⭐⭐
   - "project" 是一个通用词汇，适用于任何项目
   - 不会限制项目的发展方向

2. **谦虚务实** ⭐⭐⭐
   - 名称低调，不会给人过度设计的感觉

#### ⚠️ 缺点

1. **语义不够具体** ⭐⭐
   - "project" 太通用了，无法准确描述项目特性
   - 任何项目都可以叫 "project"

2. **不符合项目实际** ⭐⭐⭐
   - 项目包含了多个应用和基础设施，不仅仅是"一个项目"
   - 更像是一个平台或框架集合

3. **行业惯例不符** ⭐⭐⭐
   - 多模块、包含基础设施的项目通常不使用 "project"
   - Spring、Netflix 等类似项目都使用更具体的名称

## 三、行业案例参考

### 使用 "platform" 的案例

- `spring-platform` - Spring 平台
- `netflix-platform` - Netflix 平台
- `apache-platform` - Apache 平台

**特征**：包含多个组件、提供基础设施、支持多种应用

### 使用 "project" 的案例

- `spring-boot-project` - Spring Boot 项目（特定框架项目）
- `apache-maven-project` - Maven 项目（特定工具项目）

**特征**：通常是单一框架或工具的开发项目，而不是多应用平台

## 四、项目定位判断

### 当前项目特征

1. ✅ **包含多个业务应用**（OAuth Server/Client/Resource）
2. ✅ **包含基础设施组件**（幂等平台、异常处理）
3. ✅ **多模块统一管理**
4. ✅ **支持未来扩展**（可以添加更多应用和基础设施）

### 判断结论

**项目定位更像是一个"平台"而不是"项目"**

理由：
- 提供了基础设施能力（幂等、异常处理）
- 包含了多个业务应用
- 支持其他项目复用基础设施
- 有明确的平台特征

## 五、最终推荐

### 🏆 推荐：`tiny-platform` ⭐⭐⭐⭐⭐

#### 推荐理由

1. **语义最准确**
   - 准确描述了项目的实际内容和定位
   - 表明这是一个包含多个应用和基础设施的平台

2. **符合行业惯例**
   - 与 Spring、Netflix 等类似项目命名一致
   - 业界认可度更高

3. **未来扩展性好**
   - 无论增加应用还是基础设施，名称都适用
   - 不会因为项目发展而需要再次重命名

4. **专业感强**
   - 更适合企业级项目
   - 对外展示时更有说服力

5. **与子模块命名一致**
   - 已有 `tiny-idempotent-platform` 子模块
   - 命名风格统一（`tiny-xxx-platform`）

### 不推荐：`tiny-project`

**原因**：
- 语义太泛，无法准确描述项目特性
- 不符合项目实际定位（更像平台而非单一项目）
- 与行业惯例不符

## 六、实施建议

### 如果选择 `tiny-platform`

1. **重命名根目录**：`tiny_project` → `tiny-platform`
2. **更新所有 POM 的 parent 引用**
3. **统一 `tiny_web` 命名**：`tiny_web` → `tiny-web`（统一使用连字符）

### 命名一致性

```
tiny-platform/                    # 根项目（平台）
├── tiny-oauth-server            # 业务应用
├── tiny-oauth-client            # 业务应用
├── tiny-oauth-resource          # 业务应用
├── tiny-web                     # 业务应用（统一命名）
├── tiny-idempotent-platform     # 基础设施平台
├── tiny-common-exception        # 基础设施组件
└── ...
```
